## 正文

最近三个月，我一直在做一件听起来挺别扭的事：白天我是个做 AI 智能体平台的 PM，晚上写小说。结果我发现自己对市面上所有 AI 写作工具都不满意——索性自己搭了一套。

这个项目叫 **multi-agent-fiction-lab**。它不是一个装上就能用的写作工具，是一套**用多智能体辅助写长篇小说的方法论**，已经在我自己两部小说上跑通：其中一部用我**完全手写**的另一部作为"工艺基准"，来验证多智能体工作流能不能保住作家的工艺。

它的核心设计取舍是反潮流的：**人应该被设计在循环里**。

---

### 整个 AI 创作工具圈在做什么

我把市面上做 AI 写作 / AI 智能体编排的工具分两类：

**写作辅助类**：Sudowrite / NovelAI / 彩云小梦 / 妙笔 / 笔灵之类。它们做句子级流畅、风格模仿、剧情灵感。强在生成，弱在长篇连贯——超过 5 万字就开始角色漂移、情节断、文风塌。

**Agent 框架类**：LangGraph / AutoGen / CrewAI / OpenAI Swarm 之类。它们做工具编排、角色分配、结构化 I/O。强在工作流引擎，弱在创作——你能把任何任务包成 agent 调度，但 agent 的角色定义、判断标准、对"好"的认知，都得靠你自己想清楚。

**两类工具都不能直接用来写一部要扛得住 10 万字的小说**。写作辅助类没有结构判断；agent 框架类没有创作判断。

### 我做的事情

不是再做一个写作工具，是**先想清楚长篇小说要扛住什么，然后用 multi-agent 系统去守住这些**。

具体三个设计选择：

#### 1. 两支 agent 团队，不是一个流水线

教科书做法是：一个 agent 生成，一个 agent 审稿，一个 agent 修改。这是流水线思维。

我做了**两支 agent 团队**：

- **4-agent 大纲讨论团队**：发散 / 挑错 / 审美 / 收束（乌骓）。开头三个独立提案；第二轮让挑错 specifically 反驳发散，让审美做仲裁；最后乌骓出唯一裁决。
- **5-agent 正文审校团队**：节拍 / 调度 / 声线 / 禁区 / 收束。节拍先做门控（如果章节发动机不成立，不往下走）；调度和声线并行；禁区扫词面问题；最后收束合并出修改清单。

为什么是两支不是一支？因为**写大纲的思考方式跟审正文不一样**——大纲层是结构辩论，正文层是分层审计。混在一起两边都做不好。

#### 2. Memory 是工作流问题，不是基础设施问题

业界主流：用 vector DB 存设定，RAG 检索相关片段。

我的做法：**完全不用 vector DB**。每次会话的 input package 是我手动 curate 的——锁定事实表、人物档案、"不可改动"清单。

慢但对。长篇小说在做"结构性新动作"（一次转折、一次揭晓）时，vector 检索会给错的片段——因为它检索"最相似"，但你恰恰需要的是"前面没出现过的"。Curation 的成本是写一次 input package 一两个小时；vector 检索看似免费但每次取错就是隐性税。

#### 3. 工艺基准先于 AI

这条业界几乎没人做。

大多数 AI 创作项目跳过一个问题：**对作者本人来说，"好"到底是什么**？他们直接拿一堆作品训练风格，希望 agent 学会复现。但"风格"是表面现象，下面藏着的是作家的**工艺判断**——他在某些场景下绝不会写哪些东西，他在某些动作背后藏了什么物件叙事链。

我的做法：先用一部**完全手写**的长篇小说作为"工艺基准"。它的 7 条工艺红线（物件叙事链 / 双对照 / B 类记账 / 工艺红线 / "不是 X 是 X" 禁用 / 克制不写成 SOP / 章节物理钩子）被显化成方法论文档。然后另一部用多智能体工作流写的小说，要做的就是**承接这些工艺判断**——不丢三重打击、不简化复杂关系、不被概念抹平。

两部小说在仓库里是**有序进阶**关系：手写本是基准，AI 协作本是工作流验证。

### 跟 LangGraph / AutoGen / CrewAI 的关系

我自己用的编排平台叫 OpenClaw，是我从零搭的——LLM API + Lima VM gateway + 各 agent 的 SOUL.md 持久化人格。

跟另外三个框架不在同一层抽象：它们是 wrap 已有结构；OpenClaw 是 build from primitives。这本身是 portfolio 卖点（show 我能从最底层做起，不是只会用框架），但在 v0.3 框架对比的研究层我必须明示这种 asymmetry——避免评审误读成"不公平对比"。

### 为什么人在循环里

整个 AI 行业 2024-2026 都在追 autonomous orchestrator——Devin / Operator / 各种 agentic workflow。

我反方向：**让作家做信使，做 agent 间通信的载体**。Agent 们彼此不直接对话，全部通过 markdown 文件持久化讨论；下一个 session 启动时，作家把上下文拼好。

慢吗？慢。但创作判断不能完全外包。**作者品味是不可替代的变量；工作流的职责是保住它，不是替换它。**

我把这个判断写进了仓库的核心论点。如果你来招 AI PM，看到我反 autonomous 还坚持把人放进去——这是我希望你看到的。

### 一个细节：agent 也是有人格的

我特意做了一件别人不做的事：**每个 agent 在自己 SOUL.md 里写"我是谁"**——不是"你扮演 X 角色"，是"我承诺守护 X 这一件事"。差别看似小，结果不一样：

- 任务导向的 agent 会在压力下偷偷向使用者最近一次指令倾斜（你说"再短一点"它就开始砍内容）
- 身份导向的 agent 会持续守住它的承诺（即使你说"再短一点"它会先告诉你"如果再短我就要砍掉 X，这违反我的边界"）

长篇小说创作里第二种行为远比第一种重要。
